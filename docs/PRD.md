# Projects Tracker

Краткое описание: `Projects Tracker` — внутренний сервис для преподавателя, который помогает вести учет посещаемости занятий, отслеживать прогресс ученических проектов на GitHub и получать AI-оценку степени реализации по репозиторию, ТЗ и плану разработки.

## 1. Обзор продукта

`Projects Tracker` — это teacher-only сервис для учета посещаемости и контроля мини-проектов учеников на курсе по разработке приложений с ИИ.

Продукт решает две равнозначные задачи:

- учет посещаемости занятий;
- контроль проектной работы учеников через GitHub и AI-анализ.

## 2. Контекст использования

- системой пользуется один преподаватель;
- в MVP ученики не имеют доступа к системе;
- архитектура должна оставаться готовой к будущему student-access через GitHub OAuth;
- занятия проходят по вторникам, четвергам и пятницам;
- посещаемость считается в рамках недели с понедельника по пятницу;
- каждый ученик должен посетить минимум 2 занятия в неделю;
- один ученик может вести несколько проектов;
- один проект может выполняться группой из двух учеников, но в системе должен существовать один раз с общим GitHub-репозиторием и общим AI-анализом;
- репозитории хранятся только на GitHub;
- ТЗ и план разработки хранятся в Markdown;
- степень реализации проекта оценивается AI по анализу репозитория;
- преподаватель может вручную переопределять AI-оценку;
- уведомления ученикам и weekly digest преподавателю отправляются через Telegram;
- Appwrite используется как основная backend-платформа для базы данных, аутентификации и хранения.

## 3. Цели

### 3.1. Посещаемость

- дать преподавателю возможность быстро отмечать явку на каждом занятии;
- автоматически считать недельное выполнение нормы посещаемости;
- уведомлять учеников через Telegram, если за неделю они посетили меньше 2 занятий.

### 3.2. Контроль проектов

- хранить все ученические проекты в одном месте;
- сохранять метаданные проекта, Markdown ТЗ и план разработки;
- синхронизировать метаданные репозитория из GitHub;
- анализировать репозитории с помощью AI и оценивать степень реализации;
- подсвечивать проблемные случаи на дашборде.

## 4. Пользователи и роли

### 4.1. Преподаватель

Единственная активная роль в системе. Имеет полный доступ ко всем данным и действиям.

### 4.2. Ученики

В MVP не входят в систему и не имеют доступа к интерфейсу. Получают только уведомления в Telegram.

В дальнейшей версии продукта должен появиться student-access через GitHub OAuth. Связка с учеником должна строиться по `GitHub user id`, а не по `GitHub username`.

## 5. Функциональные требования

### 5.1. Модуль посещаемости

- система автоматически создает занятия на неделю по шаблону:
  - вторник;
  - четверг;
  - пятница.
- преподаватель может:
  - вручную изменить дату занятия;
  - удалить автоматически созданное занятие.
- для каждого занятия отображается список активных учеников;
- по каждому ученику ставится одна отметка:
  - был;
  - не был.
- нужны массовые действия:
  - отметить всех как `был`;
  - очистить все отметки;
  - отметить выбранных учеников.
- система считает недельную статистику посещаемости по каждому ученику:
  - сколько занятий создано за неделю;
  - сколько занятий ученик посетил;
  - выполнена ли недельная норма.
- правило недельной нормы:
  - ученик должен посетить минимум 2 занятия в интервале понедельник-пятница.
- преподаватель может просматривать историю посещаемости по ученику и по неделям;
- система должна показывать список нарушителей за неделю.

### 5.2. Модуль учеников

- создание ученика вручную;
- редактирование ученика;
- импорт учеников из XLSX.
- по ученику хранятся поля:
  - фамилия;
  - имя;
  - GitHub user id;
  - GitHub username;
  - Telegram username;
  - Telegram chat id;
  - заметка.
- карточка ученика должна показывать:
  - сводку по посещаемости;
  - список проектов;
  - последнюю активность по проектам;
  - последнюю AI-сводку;
  - teacher-only действие редактирования контактных данных, включая `Telegram chat id`.

### 5.3. Модуль проектов

- один ученик может иметь несколько проектов;
- один проект может иметь нескольких участников;
- по каждому проекту хранятся:
  - название;
  - краткое описание;
  - URL GitHub-репозитория;
  - GitHub owner;
  - GitHub repository name;
  - default branch;
  - статус проекта;
  - Markdown ТЗ;
  - Markdown план разработки;
  - дата последнего коммита;
  - SHA последнего коммита;
  - дата последней синхронизации;
  - дата последнего AI-анализа;
  - AI-степень реализации;
  - ручная степень реализации;
  - AI-сводка;
  - флаги риска.
- дополнительно для проекта должны храниться участники группы, причем ownership GitHub-репозитория и membership учеников должны быть разделены;
- поддерживаемые статусы проекта:
  - `idea`;
  - `planning`;
  - `in_progress`;
  - `review`;
  - `done`;
  - `paused`.
- преподаватель может:
  - создавать и редактировать проект;
  - добавлять второго ученика в уже существующий проект без создания дубля;
  - редактировать ТЗ и план разработки;
  - вручную запускать синхронизацию с GitHub;
  - вручную запускать AI-анализ;
  - задавать ручной override степени реализации;
  - отключать override.
- итоговая степень реализации не хранится как canonical-поле и вычисляется на лету.

### 5.4. Интеграция с GitHub

- на входе у проекта есть URL GitHub-репозитория;
- система должна извлекать и сохранять:
  - owner;
  - repo;
  - default branch;
  - README;
  - дерево репозитория;
  - последние коммиты;
  - дату последнего коммита;
  - SHA последнего коммита.
- датой последнего коммита считается самый поздний коммит в репозитории;
- issues и pull requests не входят в MVP;
- система должна концептуально анализировать весь репозиторий, но технически делать это поэтапно:
  - загрузить дерево репозитория;
  - определить релевантные файлы;
  - собрать репрезентативные исходники и markdown-артефакты;
  - передать структурированный набор данных в AI.
- флаги риска:
  - некорректный GitHub-репозиторий;
  - репозиторий пустой;
  - отсутствует ТЗ;
  - отсутствует план разработки;
  - нет коммитов 7+ дней;
  - реализация расходится с ТЗ;
  - план разработки расходится с текущим состоянием проекта.

### 5.5. AI-анализ

- входные данные AI-анализа:
  - Markdown ТЗ;
  - Markdown план разработки;
  - README;
  - дерево репозитория;
  - релевантные исходные файлы;
  - метаданные коммитов.
- выход AI-анализа:
  - краткая сводка проекта;
  - степень реализации в процентах;
  - что реализовано;
  - что частично реализовано;
  - что не реализовано;
  - оценка соответствия ТЗ;
  - оценка соответствия плану;
  - список рисков;
  - следующие шаги.
- подход к оценке степени реализации:
  - мягкая экспертная оценка;
  - без жесткого чеклистного сопоставления;
  - с учетом фактического прогресса, даже если структура проекта отличается от плана.
- AI-отчеты должны храниться как неизменяемые снимки;
- каждый AI-отчет хранит:
  - версию;
  - timestamp;
  - снимок входных данных;
  - данные о модели и конфигурации анализа;
  - сгенерированный результат.
- AI-анализ не запускается, если отсутствует ТЗ или план разработки.
- режимы AI-анализа:
  - ручной запуск;
  - запуск по расписанию;
  - запуск после обнаружения новых коммитов.

### 5.6. Ручной override

- преподаватель может вручную переопределить степень реализации;
- AI-оценка при этом не удаляется;
- итоговая степень реализации рассчитывается так:
  - если override включен, используется ручное значение;
  - иначе используется AI-оценка.
- после следующего AI-анализа ручной override автоматически отключается.
- интерфейс должен явно показывать:
  - AI-значение;
  - ручное значение;
  - итоговое значение.
- по override должны храниться:
  - кто изменил;
  - когда;
  - комментарий.

### 5.7. Telegram-уведомления

- уведомления ученикам:
  - отправлять предупреждение, если за неделю посещено меньше 2 занятий.
- дополнительно поддерживается отправка уведомлений в общий чат учеников.
- уведомления преподавателю:
  - отправлять weekly digest.
- доставка строится по Telegram chat id;
- лог уведомлений должен хранить:
  - тип;
  - получателя;
  - payload;
  - статус отправки;
  - время отправки;
  - ответ провайдера при необходимости.
- MVP-уведомление ученику:
  - короткое сообщение о недостаточной посещаемости.
- в общем чате допустимы персональные упоминания конкретных учеников.
- MVP weekly digest преподавателю:
  - только сводка;
  - без детального drill-down;
  - должен включать:
    - нарушителей по посещаемости;
    - учеников с нулевой посещаемостью за неделю;
    - проекты без коммитов 7+ дней;
    - проекты без ТЗ;
    - проекты без плана разработки;
    - проекты с низкой степенью реализации.

### 5.8. Импорт из XLSX

- система должна поддерживать импорт учеников из XLSX;
- входные колонки:
  - `Фамилия`;
  - `Имя`;
  - `Гитхаб ник`;
  - `телеграм ник`.
- импорт должен поддерживать:
  - загрузку файла;
  - preview колонок;
  - маппинг колонок;
  - валидацию строк;
  - отчет по ошибкам.
- если в файле нет Telegram chat id, преподаватель может заполнить его позже вручную.

### 5.9. Дашборд

Дашборд преподавателя должен показывать:

- текущее или ближайшее занятие;
- учеников, которым еще не проставлена отметка;
- нарушителей текущей недели;
- учеников с нулевой посещаемостью за неделю;
- проекты без коммитов 7+ дней;
- проекты без ТЗ;
- проекты без плана разработки;
- проекты с низкой степенью реализации;
- последние AI-отчеты.

Пороговые категории прогресса на дашборде:

- критический: ниже 25%;
- низкий: 25% и выше, но ниже 50%;
- средний: 50% и выше, но ниже 75%;
- высокий: 75% и выше.

## 6. Пользовательские сценарии

### 6.1. Сценарий учета посещаемости

1. Преподаватель открывает текущую неделю.
2. Система уже содержит автоматически созданные занятия на вторник, четверг и пятницу.
3. Преподаватель открывает нужное занятие.
4. Быстро отмечает посещаемость.
5. Система сохраняет данные и обновляет недельную статистику.

### 6.2. Сценарий контроля проекта

1. Преподаватель открывает карточку ученика.
2. Выбирает один из проектов ученика.
3. Видит метаданные репозитория, ТЗ, план, последний коммит и последнюю AI-оценку.
4. При необходимости запускает sync или AI-анализ.
5. Просматривает отчет.
6. При необходимости задает ручной override.

### 6.3. Сценарий weekly monitoring

1. В конце недели система рассчитывает посещаемость.
2. Система определяет нарушителей.
3. Система отправляет Telegram-уведомления ученикам.
4. Система отправляет преподавателю сводный weekly digest.

## 7. Бизнес-правила

- неделя учета посещаемости: понедельник-пятница;
- стандартный шаблон занятий на неделю: вторник, четверг, пятница;
- недельная норма посещаемости: минимум 2 занятия;
- если в неделе создано меньше 2 занятий, соответствие считается относительно фактически созданных занятий;
- посещаемость закрытой прошлой недели не должна изменяться без явного действия преподавателя;
- AI-анализ запрещен, если отсутствует ТЗ или план разработки;
- порог неактивности проекта: 7 дней без коммитов;
- порог низкой степени реализации: ниже 25%.

## 10. Фоновые задачи

Необходимые scheduled jobs:

- генерация занятий на неделю;
- синхронизация репозиториев проектов;
- запуск AI-анализа по подходящим проектам;
- расчет недельной посещаемости;
- отправка уведомлений ученикам;
- отправка weekly digest преподавателю.

Рекомендуемая периодичность:

- генерация занятий: в воскресенье вечером или в понедельник утром;
- GitHub sync: 1-2 раза в день;
- AI-анализ: по кнопке и раз в день для активных проектов;
- attendance alerts: в пятницу вечером;
- weekly digest преподавателю: в пятницу вечером после рассылки alert-уведомлений.

## 11. Нефункциональные требования

- только teacher-only защищенный доступ;
- секреты GitHub, OpenAI и Telegram должны храниться только на сервере;
- все критичные изменения должны быть аудируемыми;
- интерфейс посещаемости должен позволять быстро отмечать явку на занятии;
- система должна переживать сбои внешних интеграций без потери данных;
- для уведомлений и фоновых задач нужен retry-механизм;
- AI-запросы и результаты должны логироваться для отладки и контроля затрат.

## 12. Архитектурные рекомендации

- использовать Appwrite как основную operational backend-платформу;
- хранить canonical версии ТЗ и плана разработки в Appwrite, даже если копии лежат в репозитории;
- нормализовать идентичность GitHub-репозитория в поля `owner/repo/default branch`;
- анализ репозитория строить поэтапно, а не отправлять весь repo в AI одним сырым запросом;
- считать AI-оценку рекомендательной и хранить manual override отдельно;
- использовать Telegram chat id как ключ доставки, а не Telegram username;
- хранить недельную статистику посещаемости как снимки для быстрых дашбордов и отчетов.

## 13. MVP

- аутентификация преподавателя через GitHub OAuth;
- CRUD учеников;
- teacher-only страница редактирования ученика;
- импорт учеников из XLSX;
- автоматическая генерация занятий на неделю;
- управление занятиями и отметка посещаемости;
- расчет недельной посещаемости;
- CRUD проектов;
- редактирование Markdown ТЗ и плана разработки;
- GitHub sync с сохранением последнего коммита;
- ручной AI-анализ проекта;
- ручной override степени реализации;
- Telegram-уведомления ученикам и в общий ученический чат;
- weekly digest преподавателю в Telegram;
- дашборд с рисками по посещаемости и проектам.

## 14. Риски

- анализ полного репозитория может стать дорогим, если не ограничивать объем контекста;
- ограничения GitHub API по rate limits;
- доставка в Telegram зависит от корректности `chat_id`;
- AI-оценка степени реализации по природе субъективна;
- отсутствие ТЗ или плана разработки ухудшает качество AI-анализа.
